# Виконані завдання

1. Обмеження дескрипторів файлів (`ulimit -n`)
Досліджено вплив кількості відкритих файлів на стабільність системи.
- Встановлення занадто низького ліміту (наприклад, 10) призводить до того, що навіть базові команди (як `sudo`) перестають працювати через неможливість відкрити нові файли.

2. Робота з інструментом `perf`
Налаштовано систему для аналізу продуктивності.
- Для обходу обмежень ядра використано команду:
  `sudo sysctl -w kernel.perf_event_paranoid=-1`

3. Максимальний розмір файлу (`ulimit -f`)
Створено скрипт `dice_game.py`, який імітує кидання кубика.
- Встановлено ліміт 1 блок (1 КБ).
- Програма коректно обробила помилку `OSError: [Errno 27] File too large`, коли файл перевищив дозволений розмір.

4. Обмеження часу процесора (`ulimit -t`)
Скрипт `lottery.py` запускає нескінченний цикл генерації чисел.
- Ліміт часу ЦП — 1 секунда.
- Система автоматично завершила процес (статус `Killed`), демонструючи примусове обмеження ресурсів.

5. Утиліта копіювання з перевірками
Розроблено програму `copy_file.py`, яка перевіряє:
- Наявність аргументів (джерело та ціль).
- Доступність файлу для читання.
- Можливість запису в цільову папку.
- Перевищення лімітів під час копіювання.

6. Обмеження сегмента стеку (`ulimit -s`)
Використано рекурсію в `stack_test.py` для перевірки пам'яті стеку.
- Програма досягла внутрішнього ліміту безпеки Python, що запобігло критичному збою всієї системи.

7. Ліміт віртуальної пам'яті (`ulimit -v`)
Скрипт `memory_test.py` намагався виділити великі об'єми пам'яті.
- Ліміт: 150 МБ.
- Перехоплено `MemoryError`. Також помічено, що при такому ліміті не запускаються навіть базові редактори (як `nano`), бо вони потребують більше пам'яті для роботи.


# Висновок
Під час роботи було доведено, що ліміти `ulimit` є ефективним інструментом контролю ресурсів. Написані скрипти на Python продемонстрували здатність програмного забезпечення "м'яко" обробляти системні обмеження, не допускаючи повного зависання ОС.
